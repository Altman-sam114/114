# v0.15（StoreKit / EventKit 本地说明）

## 版本号

v0.15

## 版本分配依据

人工总目标要求持续优化 UI、分类和 CI 测试，并由 Agent X 自行寻找下一轮优化目标。v0.14 已补齐 iOS simulator destination 解析脚本并通过最新 `origin/main` 云端 artifact 验收。本轮继续在 `v0（持续优化）` 阶段递增为 v0.15，处理 `update_log.md` 和 `md/flow/flow.md` 中 StoreKit / EventKit 本地配置和验证说明不足的遗留项。

## 背景

当前 iOS 和 macOS Pro 权益都通过 StoreKit 2 实现，商品 ID 均为 `com.example.ChronoFocus.pro.analytics`。iOS 和 macOS 日历同步都通过 EventKit 读取从今天零点起 45 天范围内的非全天事件，并通过 `FocusStore.upsertExternalTask(...)` 合并为待办。README 和测试规范目前只描述功能与 CI 标记检查，缺少本地配置、人工验证、失败状态和禁止边界说明，导致后续调试 StoreKit / EventKit 时容易误以为可以直接改核心数据或伪造权益。

## 目标

- 在 README 增加 StoreKit / EventKit 本地配置与人工验证说明。
- 在 `md/test/test.md` 增加 StoreKit / EventKit 的本地人工验证边界、触发条件和不纳入默认 CI 的原因。
- 在 `update_log.md` 记录 v0.15，并把遗留项从“缺少配置说明”调整为“如需自动化 mock，可后续补充”。
- 在 `md/flow/flow.md` 的未来扩展点中同步当前状态。
- 不修改业务逻辑、不新增 mock、不新增第三方依赖。

## 非目标

- 不新增 `.storekit` 配置文件。
- 不实现 StoreKit 或 EventKit mock 服务。
- 不绕过 `PremiumAccessService` / `MacPremiumAccessService` 伪造 Pro 权益。
- 不绕过 `CalendarSyncService` / `MacCalendarSyncService` 直接写入日历任务。
- 不修改商品 ID、日历导入规则、权限申请逻辑或 UI 行为。

## 当前架构依据

- `ChronoFocus/Services/PremiumAccessService.swift` 和 `ChronoFocusMac/Services/MacPremiumAccessService.swift` 使用 StoreKit 2 的 `Product.products(for:)`、`Transaction.currentEntitlements`、`Transaction.updates`、`AppStore.sync()`。
- 两端商品 ID 均为 `com.example.ChronoFocus.pro.analytics`。
- `ChronoFocus/Services/CalendarSyncService.swift` 和 `ChronoFocusMac/Services/MacCalendarSyncService.swift` 在首次授权时使用 EventKit `requestFullAccessToEvents()`，当前实现也接受已有 `writeOnly` / `authorized` 状态，并同步从今天零点起 45 天范围内的非全天事件。
- 日历事件通过 `FocusStore.upsertExternalTask(...)` 入库，分类来自原系统日历标题，开始时间作为 due date，不自动开启番茄钟。
- `scripts/verify_project.sh` 已检查 StoreKit、EventKit、权限描述、日历同步和 `upsertExternalTask` 关键标记。

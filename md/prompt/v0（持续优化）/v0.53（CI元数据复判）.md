# v0.53（CI 元数据复判）

## 版本号

v0.53

## 版本分配依据

人工未指定小版本号；当前 `update_log.md` 和 `md/prompt/v0（持续优化）/` 最新版本为 v0.52，因此本轮自动递增为 v0.53。

## 背景

总目标要求持续优化 UI、分类体验和 CI 测试。v0.52 已补齐 Mac 任务行和小窗分类 badge 的 Voice Control 输入标签，并通过最新 `origin/main` 云端 artifact 验收。

当前 `.github/workflows/ci-results.yml` 已在 manifest 中写入 `shortSha`、`workflowName`、`projectName`、Mac/iOS scheme、destination 和 `projectSpecificReports`，artifact index 也写入 `version` 与 `createdAt`。但 `scripts/validate_ci_artifact.rb` 主要复判 branch、commit、run、attempt、路径、outcome 和本地文件元数据，对这些已有身份/元数据字段覆盖不足。错 workflow、错 project/scheme、错 short SHA 或报告列表漂移时，Agent C 的结构化脚本不一定能第一时间给出明确失败项。

人工明确要求：“不得在本地测试，都去云端”。本轮不得运行本地测试、构建、格式检查或验证脚本，最终验收只以 `origin/main` 最新 GitHub Actions artifact 为准。

## 目标

1. `scripts/validate_ci_artifact.rb` 新增 manifest 元数据复判：
   - `shortSha` 必须等于期望 commit 前 7 位。
   - `workflowName` 必须为 `ChronoFocus CI Results`。
   - `projectName` 必须为 `ChronoFocus`。
   - Mac/iOS scheme 和 destination 必须匹配 workflow 当前约定。
   - `createdAt` 必须存在且可按 ISO8601 解析。
2. 新增 artifact index 元数据复判：
   - `index["version"]` 必须与 manifest version 一致。
   - `index["createdAt"]` 必须存在且可按 ISO8601 解析。
3. 新增 `projectSpecificReports` 复判：
   - manifest 中报告名称和路径必须匹配当前 allowlist。
   - 每个报告路径必须出现在 artifact index 中。
   - 每个报告路径下载到本地后必须非空。
4. `scripts/verify_project.sh` 的小型成功 fixture 补齐上述字段，并新增负向 fixture，确认 manifest 元数据被篡改时 validator 会失败。
5. README、测试规范、核心流程文档和更新日志同步本轮 CI 复判范围。

## 非目标

- 不修改 `.github/workflows/ci-results.yml` 的产物结构、artifact 名称、上传路径或构建命令。
- 不改变 UI、模型、计时状态机、持久化、计划生成或平台服务。
- 不扩大为 artifact 签名、加密、历史 run 下载或 PR 流程。
- 不运行任何本地测试、构建、格式检查或验证脚本。

## 当前架构依据

- `.github/workflows/ci-results.yml` 已生成 `shortSha`、`workflowName`、`projectName`、`scheme`、`destination`、`macScheme`、`macDestination`、`iosScheme`、`iosDestination`、`createdAt` 和 `projectSpecificReports`。
- `scripts/validate_ci_artifact.rb` 已集中负责 Agent C 下载 artifact 后的结构化复判，适合增加元数据检查。
- `scripts/verify_project.sh` 已维护小型成功 artifact fixture 和多组负向 fixture，适合补齐 manifest 元数据错包的拒绝路径。

## 实现步骤

1. 同步 `main`：执行 `git fetch origin`、`git switch main`、`git pull --ff-only origin main`；若 `git pull` 审批超时但 `git fetch origin` 后 `HEAD == origin/main`，记录该事实并继续。
2. 修改 `scripts/validate_ci_artifact.rb`：
   - `require "time"`。
   - 新增 manifest metadata 常量，覆盖 workflow、project、scheme 和 destination。
   - 新增 project report allowlist 常量。
   - 增加 `manifest short sha`、`manifest metadata`、`manifest created at` 检查。
   - 增加 `index version`、`index created at` 检查。
   - 增加 `manifest project reports` 检查，确认 report name/path 精确匹配、路径存在于 index 且本地非空。
3. 修改 `scripts/verify_project.sh`：
   - 增加 grep/source contract，确认 validator 包含上述新检查名。
   - 小型成功 fixture 的 manifest/index 补齐 `shortSha`、`createdAt`、project/scheme/destination 和 `projectSpecificReports`。
   - 新增 `negative_manifest_metadata_fixture`，篡改 manifest 元数据并要求输出 `FAIL manifest metadata`。
4. 更新 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。
5. 只检查 diff 和 status，不运行本地测试命令。
6. 提交 `v0.53: 增强 CI 元数据复判` 并 push 到 `origin/main`。
7. Agent C 下载最新 run artifact 并按下方标准验收。

## 测试要求

本轮不运行本地测试、构建、`git diff --check`、`bash scripts/verify_project.sh`、YAML/plist lint 或 Xcode 命令，因为人工明确要求“不得在本地测试，都去云端”。

云端必须由 push 到 `origin/main` 触发 `.github/workflows/ci-results.yml`，并在最新 run artifact 中确认：

- `staticChecks`、`projectVerification`、`macBuild`、`iosBuild` 全部 success。
- `verify_project.log` 包含 `Category summary action contracts verified.`、`Category chip accessibility contracts verified.` 和 `Project structure verified.`。
- `scripts/validate_ci_artifact.rb` 对下载目录输出全 PASS，且包含 `PASS manifest metadata`、`PASS manifest project reports` 和 `PASS index version`。
- Mac/iOS build 日志包含 `** BUILD SUCCEEDED **`。

## 验收标准

- 最新 `origin/main` commit 的 GitHub Actions artifact 通过 Agent C 核对，manifest、run context、artifact index、run id、attempt 和 commit SHA 全部一致。
- 云端 `verify_project.sh` 已执行 manifest 元数据负向 fixture。
- 下载 artifact 后结构化 validator 输出全部 PASS。
- 本轮不依赖旧 run、本地输出或文字汇报冒充云端验收。

- 禁止清空或覆盖用户已有未跟踪文件。
- 禁止触碰未跟踪用户文件 `md/1.md` 和 `md/通用云端迭代升级提示词.md`。

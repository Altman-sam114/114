# v0.6 / 分类新建预填与筛选优先级提示词

## 版本号

v0.6

## 版本分配依据

人工要求 Agent X 持续优化 UI、分类和 CI。当前最新正式轮次为 v0.5，已修复 v0.4 云端 iOS 构建失败并建立 Agent X 主控循环。本轮继续在 v0.x 持续优化阶段小步推进分类可用性和测试覆盖，因此递增为 `v0.6`。

## 背景

v0.4 已提供 iOS/macOS 分类筛选和常用分类快选，但仍有两个可用性问题：选中某个分类后新增待办不会默认沿用该分类；分类筛选条会把当前范围内 0 项的分类和有任务的分类混在一起。v0.5 的云端 iOS generic build 已通过，说明分类 UI 代码路径可继续迭代。

## 目标

- iOS 日程页在选中分类后新增待办时，默认预填当前分类和匹配的预设色。
- macOS 日程详情页把待办分类筛选状态上提，选中分类时同步快速新增区域的分类和预设色。
- 分类筛选条按当前范围内任务数量排序：有任务的分类优先，数量更多的更靠前；0 项分类仍保留但排在后面。
- iOS 分类筛选 chip 和常用分类快选按钮提升到 44pt 最小高度，改善触达面积。
- CI/本地验证补充分类排序、空白分类归一和分类联动实现标记。

## 非目标

- 不新增持久化字段，不改变 `FocusTask.category` 语义。
- 不改 `TimerEngine`、计划生成、统计计算或通知逻辑。
- 不重构 `ScheduleView` / `MacScheduleDetailView` 的整体结构。
- 不改 GitHub artifact schema；仅更新 CI process version 和本地验证标记。
- 不提交未跟踪的无关文件或本地凭据。

## 当前架构依据

- 分类真实数据仍保存为 `FocusTask.category`，由 `FocusStore` 清洗并通过 `taskCategories` 提供列表。
- 常用分类元数据在 `TaskCategoryPreset`，适合增加非持久化筛选 option helper。
- iOS 分类筛选条位于 `ScheduleView.swift` 的 `TaskCategoryFilterBar`。
- macOS 分类筛选条位于 `MacScheduleDetailView.swift` 的 `MacCategoryFilterBar` 和 `MacTaskListPanelView`。
- 核心分类逻辑可用 `scripts/test_mac_core.swift` 编译验证；结构和标记由 `scripts/verify_project.sh` 锁定。

## 实现步骤

1. 共享模型：
   - 增加非持久化 `TaskCategoryFilterOption`，包含分类名、当前计数、预设元数据、图标和颜色 fallback。
   - 在 `TaskCategoryPreset` 增加 `prioritizedFilterOptions(categories:countProvider:)`，按有任务优先、数量降序、原始顺序稳定排序。
2. iOS：
   - `TaskEditorView` 增加 `initialCategory` 参数；新增任务用 `selectedCategory ?? "工作"` 初始化分类，编辑任务仍以原任务分类优先。
   - 初始化 accent 时优先使用原任务颜色，其次使用当前分类预设色，最后 fallback 到 `#3DE8C5`。
   - `TaskCategoryFilterBar` 使用 `TaskCategoryFilterOption` 排序后的列表渲染。
   - 分类 chip 和常用分类快选按钮最小高度改为 44pt。
3. macOS：
   - `MacScheduleDetailView` 持有共享 `selectedCategory`。
   - `MacTaskListPanelView` 改为 `@Binding var selectedCategory`。
   - 当选中具体分类时，同步快速新增的分类和预设色；清空筛选不覆盖用户手写分类。
   - `MacCategoryFilterBar` 使用同一排序 helper。
4. CI 和测试：
   - `scripts/test_mac_core.swift` 增加空白分类归一、默认分类顺序、预设元数据、筛选排序和 fallback 元数据断言。
   - `scripts/verify_project.sh` 增加分类排序 helper、iOS 新建预填、44pt 点击高度和 macOS binding/onChange 标记检查。
   - 若云端 Mac 快照发现原生控件黄色占位，按既有 `macSnapshotRendering` 模式补快照安全静态控件，不改变正常 App 交互；重点覆盖原生 `Picker`、`Toggle`、`Slider`、`DatePicker`、`Stepper` 和 bordered/prominent `Button`。
   - `.github/workflows/ci-results.yml` 的 `CI_PROCESS_VERSION` 更新为 v0.6，便于 Agent C artifact 识别本轮流程版本。
5. 文档：
   - 更新 README、flow、test 和 update_log 中受影响的内容。

## 验收标准

- iOS 选中“工程/复盘”等分类后新增待办，默认分类和匹配颜色正确；编辑已有任务不会被当前筛选覆盖。
- macOS 选中分类后，快速新增区域同步分类和预设色；清空筛选不覆盖手写分类。
- iOS/macOS 分类筛选条把当前范围内有任务的分类排在 0 项分类前面，数量多的优先。
- iOS 分类 chip 和常用分类快选按钮最小高度为 44pt。
- `scripts/test_mac_core.swift` 覆盖空白分类归一、预设顺序、排序 helper 和 fallback 元数据。
- 本地轻量检查通过，push 后 Agent C 核对最新 `origin/main` artifact，static/project/Mac build/iOS build 全为 `success`。

## main 同步、提交和 push 要求

Agent B 开始前必须执行：

```bash
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch
```

提交前确认当前分支是 `main`，只包含本轮相关 diff。提交信息：

```text
v0.6: 优化分类新建预填和筛选排序
```

完成本地轻量检查后执行：

```bash
git push origin main
```

push 后记录 commit SHA、workflow run id、run attempt、artifact 名称和云端结论。

## Agent C 复判要求

Agent C 必须下载最新 `origin/main` 对应 run 的 artifact 到：

```text
/private/tmp/chronofocus-c-review-<run_id>/
```

必须核对 manifest、failure summary、JUnit、`verify_project.log`、Mac/iOS build 日志、Mac/iOS `.xcresult` 和 Mac 快照。只有 manifest 的 `branch`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致，且 `staticChecksOutcome`、`projectVerificationOutcome`、`macBuildOutcome`、`iosBuildOutcome` 均为 `success`，本轮才能判定通过。

# v0.4 / 分类与 CI 首轮优化提示词

## 版本号

v0.4

## 版本分配依据

人工未指定版本号。当前 `update_log.md` 最新正式版本为 `v0.3 / 升级 main 直推云端验证流程`，本轮是在同一原型与协作验证阶段继续小步优化 UI 分类体验和 CI 覆盖，因此递增为 `v0.4`。

## 背景

ChronoFocus 已具备 iOS 主 App、Live Activity 扩展和 macOS 状态栏 App。当前任务、会话和计划都已有 `category` 字段，统计页也能按分类汇总，但日程页仍主要依赖手输分类，缺少快速筛选和常用分类入口。CI 当前通过 `.github/workflows/ci-results.yml` 上传 Agent C 可验收结果包，覆盖静态检查、`scripts/verify_project.sh` 和 `ChronoFocusMac` build，但 `update_log.md` 仍记录 iOS scheme 完整构建需要补强。

## 目标

- 优化日程 UI 的分类体验：增加常用分类快选、分类筛选和分类计数，让用户更快找到和创建任务。
- 保持共享数据边界：复用现有 `FocusTask.category`，不新增持久化字段，不绕过 `FocusStore`。
- 优化 CI 测试：在 GitHub Actions 结果包中新增 iOS `ChronoFocus` scheme 的 generic build 阶段，并把结果写入 manifest、JUnit 和 failure summary。
- 同步更新 README、测试规范、核心流程和更新日志中受影响的内容。

## 非目标

- 不重做整体视觉风格，不新增第三方依赖。
- 不改计时状态机、Live Activity 数据结构、StoreKit/EventKit 业务语义。
- 不改变 `FocusTask`、`FocusSession`、`PomodoroPlanItem` 的 Codable 持久化字段。
- 不引入 PR 流、非 `main` 分支或本地完整 Xcode build 作为默认流程。

## 当前架构依据

- 共享数据和计划逻辑位于 `ChronoFocus/Services/FocusStore.swift`。
- 任务模型和分类字段位于 `ChronoFocus/Models/AppModels.swift`。
- iOS 日程入口位于 `ChronoFocus/Views/ScheduleView.swift`。
- macOS 日程入口位于 `ChronoFocusMac/Views/MacScheduleDetailView.swift`。
- 当前项目专属验证入口是 `bash scripts/verify_project.sh`。
- 云端结果包 workflow 是 `.github/workflows/ci-results.yml`。

## 实现步骤

1. 在共享层增加非持久化分类预设或分类列表 helper，用于 iOS/macOS UI 复用；分类清洗继续由 `FocusStore` 负责。
2. iOS `ScheduleView`：
   - 在待办列表上方增加分类筛选 chips，包含“全部”和已有/预设分类。
   - 筛选结果参与当前日/周/月范围过滤。
   - `TaskEditorView` 保留手输分类，同时增加常用分类快选；选择预设分类时同步建议色。
3. macOS `MacScheduleDetailView`：
   - 快速新增区域增加常用分类快选。
   - 待办列表增加分类筛选，筛选 `store.upcomingTasks()`。
   - 快照渲染路径不能引入黄色缺失控件占位。
4. CI：
   - 在 `.github/workflows/ci-results.yml` 中新增 iOS build env、step、日志和 `.xcresult`。
   - manifest 增加 iOS scheme/destination/result/log/outcome 字段。
   - JUnit 增加 iOS build testcase。
   - failure summary 增加 iOS build 状态和日志入口。
   - Final CI status 必须把 iOS build 纳入失败判定。
5. 更新文档：
   - `README.md` 补充分类筛选/快选和 CI 覆盖。
   - `md/test/test.md` 补充云端 iOS build 阶段、结果包字段和当前基线。
   - `md/flow/flow.md` / `md/flow/flowchart.md` 补充分类 UI 流和 iOS build 云端阶段。
   - `update_log.md` 新增 v0.4 记录。

## 测试要求

本地轻量检查：

```bash
git diff --check
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
plutil -lint ChronoFocus.xcodeproj/project.pbxproj
bash scripts/verify_project.sh
```

触发原因：

- Swift UI 和共享 helper 改动需要 `scripts/verify_project.sh` 编译 Mac core tests 并渲染 Mac 快照。
- workflow 改动必须解析 YAML。
- 工程文件不应被破坏，保留 plist lint。

除非人工明确要求，不默认在本机跑完整 `xcodebuild`；新增 iOS build 由 `origin/main` 的 GitHub Actions 重验证。

## 文档更新要求

- `update_log.md` 必须记录 v0.4 日期、核心变更、关键文件、验证结果和遗留事项。
- `md/test/test.md` 必须记录新增 iOS build 的云端阶段和结果包内容。
- `md/flow/flow.md` 与 `md/flow/flowchart.md` 必须反映分类筛选/快选和 CI 阶段变化。
- `README.md` 只补用户视角功能和云端验证摘要，不写成第二份入口规范。

## 验收标准

- iOS 日程页可以按分类筛选当前时间范围内待办，新增/编辑待办时可以一键选常用分类。
- macOS 日程详情页可以按分类筛选未完成待办，快速新增能一键选常用分类。
- 分类功能不新增持久化字段，旧数据仍能正常解码。
- `.github/workflows/ci-results.yml` 的 manifest、JUnit、failure summary 和 final status 均包含 iOS build。
- `bash scripts/verify_project.sh` 通过，Mac 快照不出现缺失控件占位。
- push 到 `origin/main` 后，Agent C 下载最新 run 的 artifact，核对 `branch=main`、`commitSha`、run id、run attempt 和各阶段 outcome。

## 风险和禁止项

- 不要把分类筛选状态写入 `UserDefaults` 或模型，避免扩大持久化语义。
- 不要让 View 复制计划生成、统计或计时规则。
- 不要把 iOS 平台 API 引入 macOS target，也不要把 AppKit 引入 iOS target。
- 不要提交未跟踪的无关文件或本地凭据。
- 不要用旧 artifact 或文字描述冒充云端验收。

## main 同步、提交和 push 要求

Agent B 开始前必须执行：

```bash
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch
```

提交前必须确认当前分支是 `main`，只包含本轮相关 diff。提交信息：

```text
v0.4: 优化分类体验和 CI 覆盖
```

完成本地轻量检查后执行：

```bash
git push origin main
```

push 后记录 commit SHA、workflow run id、run attempt、artifact 名称和云端结论。

## Agent C 复判要求

Agent C 必须先完成 GitHub CLI 登录，然后下载最新 `origin/main` 对应 run 的 artifact 到：

```text
/private/tmp/chronofocus-c-review-<run_id>/
```

必须核对：

- `ci-artifact-manifest.json`
- `ci-failure-summary.md`
- `junit.xml`
- `verify_project.log`
- `xcodebuild.log`
- `ios-xcodebuild.log`
- `ChronoFocusMac.xcresult`
- `ChronoFocus-iOS.xcresult`
- `project-reports/mac-snapshots/`

只有 manifest 的 `branch`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致，且 static/project/Mac build/iOS build 均为 `success`，本轮才能判定通过。

# v0.65（CI版本复判）

## 版本号

v0.65

## 版本分配依据

人工总目标要求持续优化 UI 易用性、分类体验和 CI 测试。v0.64 已完成 iOS 新增入口分类上下文并通过云端 artifact 验收；本轮继续按小版本递增到 v0.65，聚焦 CI artifact 结构化复判对 process version 的固定校验。

## 背景

`.github/workflows/ci-results.yml` 使用 `CI_PROCESS_VERSION: v0.10` 生成 artifact 名称、manifest 和 artifact index。当前 `scripts/validate_ci_artifact.rb` 会核对 manifest 与 index 内部字段一致性，但没有显式确认它们等于当前预期 process version。旧版本结果包如果内部自洽，可能被结构化 validator 放行。

## 目标

- 在 `scripts/validate_ci_artifact.rb` 中新增固定预期版本常量，例如 `EXPECTED_CI_PROCESS_VERSION = "v0.10"`。
- validator 新增 `ci process version` 检查，要求 manifest 和 artifact index 的 `version` 都等于当前预期版本。
- artifact name 预期值使用固定版本常量，不再直接信任 manifest 中的 `version`。
- `scripts/verify_project.sh` 增加 validator 常量和 PASS 名称检查。
- `scripts/verify_project.sh` 增加旧 process version 负向 fixture，确保 manifest/index/summary/run context artifactName 同步降为 `v0.09` 时 validator 输出 `FAIL ci process version`。
- 同步 README、测试规范、核心流程、流程图和更新记录。

## 非目标

- 不修改 `.github/workflows/ci-results.yml` 的 `CI_PROCESS_VERSION`，本轮只复判当前 `v0.10`。
- 不改变 artifact 文件清单、构建步骤、JUnit 生成逻辑或失败摘要格式。
- 不运行本地测试、build、lint、`git diff --check` 或脚本验证；人工已要求“不得在本地测试，都去云端”。

## 当前架构依据

- `.github/workflows/ci-results.yml` 是云端结果包生成入口，当前 `CI_PROCESS_VERSION` 为 `v0.10`。
- `scripts/validate_ci_artifact.rb` 是 Agent C 下载 artifact 后的结构化复判脚本。
- `scripts/verify_project.sh` 用小型成功 fixture 和负向 fixture 锁定 validator 放行/拒绝行为。

## 实现步骤

1. 同步最新 `main`，确认当前分支为 `main`，没有无关 tracked diff；不要读取或提交用户未跟踪文件。
2. 修改 `scripts/validate_ci_artifact.rb`：
   - 增加 `EXPECTED_CI_PROCESS_VERSION = "v0.10"`。
   - 将 expected artifact name 中的版本段改为该常量。
   - 增加 `check(checks, "ci process version")`，要求 `manifest["version"]` 和 `index["version"]` 都等于常量。
3. 修改 `scripts/verify_project.sh`：
   - 检查 validator 包含 `EXPECTED_CI_PROCESS_VERSION` 和 `ci process version`。
   - 增加 `stale_process_version_fixture`，把成功 fixture 的 manifest/index version、summary Version 和 run context artifactName 同步改成 `v0.09`，要求 validator 输出 `FAIL ci process version`。
4. 同步 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。

## 测试要求

- 本轮不运行本地测试、build、lint、`git diff --check` 或脚本验证，因为人工明确要求“不得在本地测试，都去云端”。
- 提交并 push 后以 `.github/workflows/ci-results.yml` 的云端结果为准。
- Agent C 下载最新 artifact 后运行：

```bash
ruby scripts/validate_ci_artifact.rb /private/tmp/chronofocus-c-review-<run_id> \
  --commit <origin-main-sha> \
  --run-id <run_id> \
  --attempt <run_attempt>
```

- 复判输出必须包含 `PASS ci process version`。

- `md/flow/flowchart.md` 同步 artifact validator 节点。

## 验收标准

- validator 不再信任 manifest 自带 version 生成 artifact name。
- 旧 process version artifact 即使 manifest/index/run context/summary 内部同步，也会被 `FAIL ci process version` 拒绝。
- 云端 CI 成功，artifact manifest 的 `branch=main`、`commitSha`、run id、attempt 与最新 `origin/main` 一致。
- Agent C 下载最新 artifact 后复判全 PASS，并确认新增 `PASS ci process version`。

## 风险和禁止项

- 后续 bump `CI_PROCESS_VERSION` 时必须同步 validator 常量和相关文档，这是显式契约成本。
- 禁止改变 workflow 版本号或 artifact 生成流程。
- 禁止提交未跟踪用户文件。
- 禁止用旧 run、旧 artifact 或本地输出冒充云端验收。

## main 提交、push 和云端验收

- 本轮固定使用 `main`。
- 提交信息：`v0.65: 强化CI版本复判`。
- push 到 `origin/main` 后查询 `ChronoFocus CI Results` 最新 run。
- Agent C 下载最新 run 的 artifact 到 `/private/tmp/chronofocus-c-review-<run_id>/`。
- Agent C 必须核对 manifest、artifact index、run context、artifact 名称、JUnit、failure summary、`verify_project.log`、Mac/iOS build 日志、`.xcresult` 和 Mac 快照产物。

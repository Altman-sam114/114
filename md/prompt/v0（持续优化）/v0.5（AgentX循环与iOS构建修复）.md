# v0.5 / Agent X 循环与 iOS 构建修复提示词

## 版本号

v0.5

## 版本分配依据

人工以 `agentx` 指定持续优化总目标。当前 `origin/main` 最新正式提交为 `v0.4: 优化分类体验和 CI 覆盖`，但 Agent C 已从云端 artifact 确认 v0.4 首个 run 的 iOS generic build 失败。本轮是在 v0.4 基础上追加修复并建立 Agent X 主控循环文档基线，因此递增为 `v0.5`。

## 背景

ChronoFocus 现在有 iOS 主 App、iOS Live Activity 扩展和 macOS 状态栏 App。v0.4 已完成第一轮分类 UI 和 CI 覆盖优化：新增常用分类、分类筛选，以及云端 `ChronoFocus` iOS generic build。Agent C 下载 v0.4 云端结果包后确认 manifest 匹配最新 `origin/main`，但 iOS build 失败，日志指向 `ChronoFocus/Views/ScheduleView.swift` 的 `taskCount(in:)` 分类分支缺少 `return`。

人工当前总目标是由 Agent X 持续调度 A/B/C 子轮次：继续优化 UI、让分类更方便使用、优化 CI 测试，并由 Agent X 自行寻找下一轮优化目标。

## 目标

- 修复 v0.4 云端 iOS build 暴露的 Swift 编译错误。
- 将 Agent X 主控循环写入入口文档、流程文档、测试规范和 prompt 规范，明确它不能替代 Agent A/B/C，也不能跳过最新 artifact 验收。
- 记录当前 v0.5 轮次，确保后续可继续从 Agent X 拆分下一轮 UI/CI 优化。
- 保持本轮改动小而可验证，不扩大到新的 UI 功能实现。

## 非目标

- 不新增新的持久化字段、模型字段或业务规则。
- 不重构 `ScheduleView`、`FocusStore`、`TimerEngine` 或 CI workflow。
- 不改变 v0.4 已实现的分类筛选行为。
- 不提交未跟踪的无关文档、本地凭据、历史缓存或 artifact。
- 不宣称总目标完成；v0.5 通过后继续下一轮。

## 当前架构依据

- `TimerEngine` 是唯一计时状态机；本轮不触碰计时逻辑。
- `FocusStore` 是任务、设置、计划、会话和分类列表的数据边界；本轮不新增数据语义。
- iOS 日程页位于 `ChronoFocus/Views/ScheduleView.swift`，v0.4 的分类计数 helper 属于 View 层筛选展示逻辑。
- 默认云端验证入口是 `.github/workflows/ci-results.yml`，其中 iOS `ChronoFocus` generic build 是 v0.4 新增阶段。
- Agent X 只做调度判断；每轮仍必须经过 Agent A 提示词、Agent B 实现/push、Agent C artifact 验收。

## 实现步骤

1. 修复 `ScheduleView.taskCount(in:)`：
   - `category == nil` 分支继续返回当前时间范围内任务数量。
   - `category != nil` 分支显式 `return store.tasks.filter { ... }.count`。
2. 更新 Agent X 规则：
   - `AGENTS.md` 增加 `agentx` / `x:` / `X:` 入口、身份标识、职责、停止条件和禁止项。
   - `md/prompt/README.md` 增加 Agent X 不能替代 Agent A 提示词的规则。
   - `md/test/test.md` 增加 Agent X 每轮仍必须由 Agent C 核对最新 artifact 的验证规则和下载容量限制。
   - `md/flow/flow.md` 与 `md/flow/flowchart.md` 增加 Agent X 主控循环说明。
   - `README.md` 简短说明 Agent X 的使用方式。
3. 更新 `update_log.md`：
   - 记录 v0.5 版本、日期、核心变更、关键文件、验证结果和遗留事项。
   - 明确 v0.4 首个云端 run 的 iOS build 失败需要本轮修复后重新云端验证。
4. 本地轻量检查通过后，按 `v0.5: 启动 Agent X 循环并修复 iOS 构建` 提交并 push 到 `origin/main`。
5. Agent C 下载最新 run artifact，核对 manifest、JUnit、failure summary、Mac/iOS 日志、Mac/iOS `.xcresult` 和 Mac 快照。

## 测试要求

本地轻量检查：

```bash
git diff --check
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
plutil -lint ChronoFocus.xcodeproj/project.pbxproj
bash scripts/verify_project.sh
```

触发原因：

- Swift 编译修复需要至少跑项目专属 Smoke 验证。
- Agent X 和测试规则改动需要 `git diff --check`。
- workflow 虽未改动，但本轮依赖云端 CI 结果包，保留 YAML 解析作为轻量确认。

不默认跑完整本机 iOS build；最终 iOS 结论以 `origin/main` 的 GitHub Actions iOS generic build 和 Agent C artifact 核对为准。

- v0.5 提示词存在于 `md/prompt/v0（持续优化）/`，覆盖目标、非目标、实现步骤、测试、文档、验收和风险。

- push 后 Agent C 下载最新 `origin/main` 对应 artifact，manifest 的 `branch`、`commitSha`、`runId`、`runAttempt` 完全一致，且 static/project/Mac build/iOS build 均为 `success`。

## main 同步、提交和 push 要求

Agent B 开始前必须执行：

```bash
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch
```

提交前必须确认当前分支是 `main`，只包含本轮相关 diff。提交信息：

```text
v0.5: 启动 Agent X 循环并修复 iOS 构建
```

完成本地轻量检查后执行：

```bash
git push origin main
```

push 后记录 commit SHA、workflow run id、run attempt、artifact 名称和云端结论。


- `ci-artifact-manifest.json`
- `ci-failure-summary.md`
- `junit.xml`

- `xcodebuild.log`
- `ios-xcodebuild.log`

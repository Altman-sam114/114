# v0.38（CI JUnit outcome 复判）

## 版本号

v0.38

## 版本分配依据

人工未指定小版本号；当前 `update_log.md` 和 `md/prompt/v0（持续优化）/` 最新版本为 v0.37，因此本轮自动递增为 v0.38。

## 背景

总目标要求持续优化 UI、分类体验和 CI 测试。当前 validator 已核对 JUnit 四个 testcase 名称和日志入口，也会核对 manifest、run context、artifact index、failure summary、static-checks、Xcode 版本、分类可访问 contract 和 build 成功标记；但 JUnit `system-out` 中的 `outcome=` 尚未与 manifest 对应阶段 outcome 交叉验证。补齐后，Agent C 能发现 JUnit 摘要被旧值、错值或不一致阶段状态污染的结果包。

人工明确要求：“不得在本地测试，都去云端”。本轮不得运行本地测试、构建、格式检查或验证脚本，最终验收只以 `origin/main` 最新 GitHub Actions artifact 为准。

## 目标

1. `scripts/validate_ci_artifact.rb` 新增 `EXPECTED_JUNIT_OUTCOMES`，映射 JUnit testcase 到 manifest outcome key。
2. `scripts/validate_ci_artifact.rb` 新增 `junit testcase outcomes` 复判，要求每个 `system-out` 包含与 manifest 一致的 `outcome=`。
3. `scripts/verify_project.sh` 增加 marker 检查，锁定新增 validator 规则。
4. `scripts/verify_project.sh` 增加错误 JUnit outcome 负向 fixture，确认 validator 会拒绝错包。
5. README、测试规范、核心流程文档和更新日志同步本轮 CI 覆盖范围。

## 非目标

- 不改变 GitHub Actions workflow、artifact 文件列表或 CI process version。
- 不改变 SwiftUI UI、业务逻辑、数据模型、计时状态机或平台服务。
- 不改变 JUnit 生成格式，只增强下载 artifact 后的复判。
- 不运行任何本地测试、构建或验证命令。

## 当前架构依据

- `.github/workflows/ci-results.yml` 会在 `junit.xml` 的四个 testcase `system-out` 中写入 `outcome=<阶段状态>; log=<日志路径>`。
- `scripts/validate_ci_artifact.rb` 已读取 manifest 和 JUnit，并已检查 testcase 名称与日志入口，适合在同一区块增加 outcome 交叉复判。
- `scripts/verify_project.sh` 已有小型成功 fixture 和多个负向 fixture，可在正向 validator 通过后追加 JUnit outcome 篡改负向样例。

## 实现步骤

1. 同步 `main`：执行 `git fetch origin`、`git switch main`、`git pull --ff-only origin main`。
2. 在 `scripts/validate_ci_artifact.rb` 中新增 `EXPECTED_JUNIT_OUTCOMES`。
3. 在 JUnit 日志入口检查后新增 `junit testcase outcomes`。
4. 在 `scripts/verify_project.sh` 中增加 `EXPECTED_JUNIT_OUTCOMES` 和 `junit testcase outcomes` marker。
5. 在 `scripts/verify_project.sh` 的 fixture 区新增错误 JUnit outcome 负向 fixture，并断言输出 `FAIL junit testcase outcomes`。
6. 更新 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。
7. 只检查 diff 和 status，不运行本地测试命令。
8. 提交 `v0.38: 复判 CI JUnit outcome` 并 push 到 `origin/main`。
9. Agent C 下载最新 run artifact 并按下方标准验收。

## 验收标准

- 最新 `origin/main` commit 的 GitHub Actions artifact 通过 Agent C 核对，manifest、run context、artifact index、run id、attempt 和 commit SHA 全部一致。
- 下载 artifact 的 validator 输出包含 `PASS junit testcase outcomes`。
- 本轮不依赖旧 run、本地输出或文字汇报冒充云端验收。

- 禁止新增依赖具体 Xcode 版本或 runner 文案的检查。
- 禁止扩大到 workflow 重构、artifact 版本升级或 UI 改动。

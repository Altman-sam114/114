# v0.19（CI结果包负向 fixture）

## 版本号

v0.19

## 版本分配依据

人工总目标要求持续优化 UI、分类和 CI 测试，并由 Agent X 自行寻找下一轮优化目标。v0.18 已把 `ci-run-context.txt` 和 artifact 名称纳入结果包复判，并通过最新 `origin/main` 云端 artifact 验收。本轮继续优化 CI 测试质量，递增为 v0.19，给 validator 增加负向 fixture，确认错误结果包会被拒绝。

## 背景

`scripts/verify_project.sh` 当前会构造一个小型成功 artifact fixture，并运行 `scripts/validate_ci_artifact.rb` 确认 PASS。v0.18 新增 run context 身份和 artifact 名称复判后，成功路径已覆盖，但还没有证明 artifactName 错误时 validator 会失败。本轮增加一个低成本负向 fixture，避免后续误改让 validator 对错包放行。

## 目标

- 在 `scripts/verify_project.sh` 中新增 run context artifactName 错误的负向 fixture。
- 负向 fixture 必须复用已构造的小型 artifact 结构或用最低成本复制，避免新增大文件。
- `scripts/validate_ci_artifact.rb` 对负向 fixture 必须返回非 0；如果意外成功，`verify_project.sh` 必须失败。
- 负向检查应捕捉 `run context artifact name` 的 FAIL 输出，证明失败原因是本轮要覆盖的字段。
- README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`update_log.md` 同步“validator 成功和负向 fixture”覆盖范围。

## 非目标

- 不修改 `.github/workflows/ci-results.yml`。
- 不修改 validator 的复判 contract，除非发现当前错误信息不足以测试。
- 不新增大体积 artifact、DerivedData 或历史结果包下载。
- 不改 App UI、模型、计时、StoreKit、EventKit 或持久化逻辑。

## 当前架构依据

- `scripts/verify_project.sh` 已用 Python 构造小型 artifact fixture，包含 manifest、artifact index、run context、JUnit、summary、日志、Mac/iOS `.xcresult` 占位目录和 5 张小型快照。
- `scripts/validate_ci_artifact.rb` 输出固定的 `PASS/FAIL <check name>` 行，并已新增 `run context artifact name` 检查。
- bash 脚本启用了 `set -euo pipefail`，负向命令必须用 `if ...; then ... else ... fi` 或临时 `set +e` 安全处理。

## 实现步骤

1. 更新 `scripts/verify_project.sh`：
   - 在成功 fixture 通过后，复制该 fixture 到一个临时负向目录，或直接改写同一 fixture 的 `ci-run-context.txt` 后再恢复/删除。
   - 将负向 fixture 的 `artifactName` 改为明显错误值，例如 `chronofocus-ci-v0.10-main-wrong-run12345-attempt1`。
   - 运行 validator，将输出写入临时文件。
   - 如果 validator 对负向 fixture 返回 0，则打印错误并 `exit 1`。
   - 如果 validator 返回非 0，则检查输出中包含 `FAIL run context artifact name`。
   - 清理临时目录和临时输出文件。
   - 增加 marker 检查：`FAIL run context artifact name` 或 `negative_artifact_fixture` 这类稳定字符串。
2. 文档同步：
   - README 补充本地验证会用小型 fixture 覆盖 validator 成功与负向路径。
   - `md/test/test.md` 的 smoke 基线和结果包复判说明补充负向 fixture。
   - `md/flow/flow.md` / `md/flow/flowchart.md` 同步 CI 复判脚本覆盖错误包拒绝。
   - `update_log.md` 追加 v0.19 记录。

## 文档更新要求

- README 面向用户，说明项目验证覆盖 validator 成功 fixture 与错误 artifactName 负向 fixture。
- `md/test/test.md` 更新 smoke 当前基线和 Agent C 复判辅助脚本说明。
- `md/flow/flow.md` 和 `md/flow/flowchart.md` 只描述当前真实 CI 验收流。
- `update_log.md` 记录版本、日期、核心变更、关键文件、验证结果和遗留事项。

## 验收标准

- `bash scripts/verify_project.sh` 成功路径会先验证小型 artifact PASS，再验证 artifactName 错误 fixture 被 validator 拒绝。
- 若临时把负向 fixture 的 artifactName 改回正确值，脚本应因“负向 fixture 意外通过”而失败。
- push 到 `origin/main` 后，Agent C 使用最新 v0.19 run artifact 复判通过，manifest、artifact index、run context、JUnit、failure summary、日志、Mac/iOS `.xcresult` 和 Mac 快照 manifest 均一致。

## 风险和禁止项

- 禁止让负向 fixture 在 `set -e` 下提前中断脚本而无法检查失败原因。
- 禁止新增大文件、历史 artifact 下载或无关缓存。
- 禁止修改 App 业务逻辑或 UI。
- 禁止把本地负向 fixture 输出冒充云端 artifact 验收。
- 禁止读取、提交或删除无关未跟踪文件。

## GitHub Actions 和 Agent C 复判要求

- 触发 workflow：`.github/workflows/ci-results.yml`
- Agent C 下载 artifact 前必须先完成 `gh auth login`，并确认 GitHub 账号为项目授权账号。
- Agent C 下载最新 `origin/main` commit 对应 run 的 artifact 到 `/private/tmp/chronofocus-c-review-<run_id>/`。
- Agent C 运行 `ruby scripts/validate_ci_artifact.rb /private/tmp/chronofocus-c-review-<run_id> --commit <sha> --run-id <run_id> --attempt <attempt>`。
- 同时继续人工核对 manifest、artifact index、run context、JUnit、failure summary、主日志、Mac/iOS `.xcresult` 和 Mac 快照 manifest。

# v0.36（CI failure summary 身份复判）

## 版本号

v0.36

## 版本分配依据

人工未指定小版本号；当前 `update_log.md` 和 `md/prompt/v0（持续优化）/` 最新版本为 v0.35，因此本轮自动递增为 v0.36。

## 背景

总目标要求持续优化 UI、分类体验和 CI 测试。当前 validator 已核对 manifest、run context、artifact index、JUnit、static-checks、Xcode 版本、分类可访问 contract 和 build 成功标记；failure summary 目前只确认“全部通过”文案和日志入口。补齐 summary 的身份字段与阶段 outcome 复判后，Agent C 能更明确地发现 summary 来自旧 run、错 commit 或阶段状态不一致的 artifact。

人工明确要求：“不得在本地测试，都去云端”。本轮不得运行本地测试、构建、格式检查或验证脚本，最终验收只以 `origin/main` 最新 GitHub Actions artifact 为准。

## 目标

1. `scripts/validate_ci_artifact.rb` 新增 `failure summary identity` 复判。
2. `failure summary identity` 要求 summary 中 version、branch、commit、run attempt 与 manifest/CLI 参数一致。
3. `scripts/validate_ci_artifact.rb` 新增 `failure summary outcomes` 复判。
4. `failure summary outcomes` 要求 summary 中四个阶段 outcome 与 manifest 对应字段一致。
5. `scripts/verify_project.sh` 锁定 validator 必须包含上述两个复判 marker。
6. README、测试规范、核心流程文档和更新日志同步本轮 CI 覆盖范围。

## 非目标

- 不改变 GitHub Actions workflow 阶段顺序。
- 不改变 artifact 文件列表或 CI process version。
- 不改变 SwiftUI UI、业务逻辑、数据模型、计时状态机或平台服务。
- 不运行任何本地测试、构建或验证命令。

## 当前架构依据

- `.github/workflows/ci-results.yml` 会用 manifest 字段生成 `ci-failure-summary.md` 的 version、branch、commit、run attempt 和四个阶段 outcome。
- `scripts/validate_ci_artifact.rb` 已读取 manifest、summary 和 CLI 期望参数，适合在现有 summary 检查后增加交叉复判。
- `scripts/verify_project.sh` 的小型成功 fixture 已写入 summary 身份字段和 success outcome，可覆盖 validator 正向路径。

## 实现步骤

1. 同步 `main`：执行 `git fetch origin`、`git switch main`、`git pull --ff-only origin main`。
2. 在 `scripts/validate_ci_artifact.rb` 中新增 `EXPECTED_SUMMARY_OUTCOMES`，映射 summary 阶段名到 manifest outcome key。
3. 在 failure summary 检查后新增 `failure summary identity` 和 `failure summary outcomes`。
4. 在 `scripts/verify_project.sh` 中增加 marker 检查，锁定 validator 包含两个新复判项。
5. 更新 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。
6. 只检查 diff 和 status，不运行本地测试命令。
7. 提交 `v0.36: 复判 failure summary 身份` 并 push 到 `origin/main`。
8. Agent C 下载最新 run artifact 并按下方标准验收。

## 测试要求

本轮不运行本地测试、构建、`git diff --check`、`bash scripts/verify_project.sh`、Ruby validator、YAML/plist lint 或 Xcode 命令，因为人工明确要求“不得在本地测试，都去云端”。

云端必须由 push 到 `origin/main` 触发 `.github/workflows/ci-results.yml`，并在最新 run artifact 中确认：

- `staticChecks`、`projectVerification`、`macBuild`、`iosBuild` 全部 success。
- `scripts/validate_ci_artifact.rb` 对下载目录输出全 PASS，且包含 `PASS failure summary identity` 和 `PASS failure summary outcomes`。
- `verify_project.log` 包含 `Category chip accessibility contracts verified.` 和 `Project structure verified.`。
- Mac/iOS build 日志包含 `** BUILD SUCCEEDED **`。
- JUnit 仍为 4 个 testcase 且 failures 为 0。

## 文档更新要求

- README 说明 artifact validator 会复判 failure summary 身份字段和阶段 outcome。
- `md/test/test.md` 同步 validator 覆盖范围和通过证据。
- `md/flow/flow.md` 与 `md/flow/flowchart.md` 同步 CI 结果包复判链路。
- `update_log.md` 新增 v0.36 记录，说明未跑本地测试及原因。

## 验收标准

- 最新 `origin/main` commit 的 GitHub Actions artifact 通过 Agent C 核对，manifest、run context、artifact index、run id、attempt 和 commit SHA 全部一致。
- 下载 artifact 的 validator 输出包含 `PASS failure summary identity` 和 `PASS failure summary outcomes`。
- 本轮不依赖旧 run、本地输出或文字汇报冒充云端验收。

## 风险和禁止项

- 禁止用本地输出或旧 artifact 代替最新云端验收。
- 禁止新增依赖具体 Xcode 版本或 runner 文案的检查。
- 禁止扩大到 workflow 重构、artifact 版本升级或 UI 改动。
- 禁止触碰未跟踪用户文件 `md/1.md` 和 `md/通用云端迭代升级提示词.md`。

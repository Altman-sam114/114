# v0.8（Mac 快捷入口与 CI 错误摘录）

## 版本号

v0.8

## 版本分配依据

人工总目标仍是“优化 UI，更方便使用，优化分类；优化 CI 测试；自行寻找目标不断优化”。v0.7 已完成分类筛选摘要和快照清单，本轮在同一 `v0（持续优化）` 阶段继续递增为 v0.8。

## 背景

当前 macOS 小窗快捷面板中的“日程 / 统计 / 设置”按钮都调用同一个 `openDetails()`，打开详细窗口后默认停留在计时页。用户从小窗进入某个功能时还要再点侧边栏，路径不够直接。

当前 GitHub Actions 的 `ci-failure-summary.md` 会列出阶段状态和日志入口，但失败时缺少关键错误摘录。Agent C 或人工需要打开完整日志才能快速定位 Swift 编译错误、快照占位、脚本异常或 build 失败。

## 目标

- macOS 小窗快捷面板的“日程 / 统计 / 设置”分别打开或切换到对应详情 Tab。
- 如果详情窗口已经打开，再从小窗点击快捷入口时，应复用现有窗口并更新选中 Tab。
- 右键菜单“打开详细界面”和环境变量 `CHRONOFOCUS_MAC_OPEN_DETAILS` 仍默认打开计时 Tab。
- CI failure summary 在有失败阶段时追加有限的 `Failure Excerpts`，从对应日志中摘录关键错误行。
- 将 `.github/workflows/ci-results.yml` 的 `CI_PROCESS_VERSION` 更新为 v0.8。

## 非目标

- 不修改 `TimerEngine`、`FocusStore`、共享模型或持久化字段。
- 不重构 Mac 详情页布局，不改变四个详情页内部业务行为。
- 不新增第三方依赖，不引入新的 CI job。
- 不把完整日志复制进 summary，只摘录有限行数，避免 artifact 和 Step Summary 过大。

## 当前架构依据

- `TimerEngine` 仍是唯一计时状态机，本轮不新增计时状态。
- `FocusStore` 仍是核心数据仓库，本轮不新增持久化。
- Mac 状态栏和窗口管理集中在 `ChronoFocusMac/App/MacStatusBarController.swift`。
- Mac 详情页入口为 `ChronoFocusMac/Views/MacDetailView.swift`，当前 `MacDetailSection` 驱动侧边栏和详情内容。
- 小窗快捷面板在 `ChronoFocusMac/Views/MacMiniTimerView.swift`。
- 云端结果包由 `.github/workflows/ci-results.yml` 生成，`ci-failure-summary.md`、`junit.xml` 和 manifest 是 Agent C 验收入口。

## 实现步骤

1. 将 `MacDetailSection` 调整为可被 `MacStatusBarController` 和 `MacMiniTimerView` 使用的内部类型，不保持 `private`。
2. 新增可观察的详情页选择状态，例如 `MacDetailSelection`，由 `MacStatusBarController` 持有并传入 `MacDetailView`。
3. 将 `MacDetailView` 从内部 `@State` 改为使用外部 selection；默认 selection 为 `.timer`。
4. 将 `MacStatusBarController.showDetails()` 扩展为 `showDetails(section: MacDetailSection = .timer)`：
   - 每次调用先更新 selection。
   - 若窗口已存在，直接 `makeKeyAndOrderFront` 并激活 App。
   - 若窗口不存在，创建窗口并传入同一个 selection。
5. 将 `MacMiniTimerView` / `MacMiniQuickPanelView` 的 `openDetails` 改为接收 `MacDetailSection`。
6. 小窗快捷面板按钮分别调用 `openDetails(.schedule)`、`openDetails(.analytics)`、`openDetails(.settings)`；右键菜单和启动环境变量不变。
7. 在 workflow 的 Python summary 逻辑中新增失败摘录函数：
   - 只在对应阶段 outcome 非 `success` 时读取该阶段日志。
   - 匹配 `error:`、`BUILD FAILED`、`SnapshotError`、`missing-control`、`Traceback`、`Exception`、`fatal error` 等关键行。
   - 每个阶段最多摘录固定数量行，例如 8 行；每行截断到合理长度，例如 220 字符。
   - 日志缺失时输出明确的缺失提示。
8. 更新 `scripts/verify_project.sh`：
   - 检查 `showDetails(section:`、`MacDetailSelection`、`openDetails(.schedule)`、`openDetails(.analytics)`、`openDetails(.settings)`。
   - 检查 workflow 中存在 `Failure Excerpts` 和错误摘录关键字。
9. 更新 README、`md/flow/flow.md`、`md/test/test.md`、`update_log.md` 中受影响的当前事实。

## 测试要求

本地轻量检查：

```bash
git diff --check
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
plutil -lint ChronoFocus.xcodeproj/project.pbxproj
```

触发原因：

- 修改 SwiftUI/macOS 窗口入口，需要 `verify_project.sh` 编译 Mac 快照脚本并渲染小窗/详情页。
- 修改 workflow，需要 YAML 解析。
- 修改验证脚本和 CI 结果包逻辑，需要项目专属验证脚本兜底。
- 完整 Mac/iOS Xcode build 默认交给 `origin/main` 的 GitHub Actions。

## 验收标准

- 小窗快捷面板三个按钮源码分别调用 `.schedule`、`.analytics`、`.settings`。
- 已打开详情窗口时再次调用 `showDetails(section:)` 会更新 selection 并复用窗口。
- 右键菜单和 `CHRONOFOCUS_MAC_OPEN_DETAILS` 仍默认打开 `.timer`。
- `ci-failure-summary.md` 在失败阶段存在时包含 `Failure Excerpts`，并按阶段列出有限关键行；成功时仍保留 `All CI stages passed.`。
- `junit.xml` 仍包含 4 个 testcase，manifest 的 run/commit/outcome 字段不退化。
- 本地轻量检查通过。
- push 到 `origin/main` 后，Agent C 下载最新 run 的 artifact，核对 `branch=main`、`commitSha`、run id、run attempt、所有 outcome、failure summary、JUnit、日志、`.xcresult`、Mac 快照和快照 manifest。

## 风险和禁止项

- 禁止为了快捷入口重写 Mac 详情页导航结构。
- 禁止在 View 中新增计时状态或持久化状态。
- 禁止把 CI 完整日志复制进 summary。
- 禁止跳过云端 artifact 验收。
- 禁止读取、提交或删除无关未跟踪文件。
- 若云端失败，不回滚；在 `main` 上追加修复 commit 后重新 push。

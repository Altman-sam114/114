# v0.24（CI 索引身份负向 fixture）

## 版本号

v0.24

## 版本分配依据

人工总目标要求持续优化 UI、分类和 CI 测试，并由 Agent X 自行寻找下一轮优化目标。v0.23 已补齐本地缺失产物负向 fixture，并通过最新 `origin/main` 云端 artifact 验收。本轮继续收紧 CI 结果包校验覆盖，递增为 v0.24。

## 背景

`scripts/validate_ci_artifact.rb` 已检查 manifest、run context 和 artifact index 的 branch、commit、run id、attempt 身份字段，也检查下载后本地产物是否存在且非空。`scripts/verify_project.sh` 当前已有小型成功 fixture、artifactName mismatch 负向 fixture 和本地缺失产物负向 fixture，但还缺少“artifact index 本身来自旧提交或错误 run”的负向覆盖。本轮用最小 fixture 证明 validator 会拒绝 index 身份错包。

## 目标

- 在 `scripts/verify_project.sh` 的 validator fixture 中新增 artifact index 身份不一致负向 fixture。
- 负向 fixture 复制成功 fixture 后只篡改 `ci-artifact-index.json` 的 `commitSha`。
- 运行 `scripts/validate_ci_artifact.rb` 时必须失败。
- 显式匹配 `FAIL index commit`，确认失败来自 artifact index 身份复判。
- README、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、`update_log.md` 同步当前覆盖范围。

## 非目标

- 不修改 GitHub Actions workflow 产物格式。
- 不修改 validator 的真实验收规则。
- 不修改 Swift UI、模型、计时状态机、平台服务或 Xcode target。
- 不引入体积阈值、历史 artifact 下载或大体积测试数据。

## 当前架构依据

- `scripts/validate_ci_artifact.rb` 已有 `index commit`、`index run`、`index attempt` 等身份检查。
- `scripts/verify_project.sh` 已能生成完整小型 artifact fixture，并运行 validator 成功路径。
- v0.19、v0.23 已用负向 fixture 锁定 artifactName mismatch 和本地缺失产物两条拒绝路径，本轮沿用同样模式增加 index 身份错包拒绝路径。

## 实现步骤

1. 更新 `scripts/verify_project.sh`：
   - 在 artifactName mismatch 负向 fixture 后新增 `negative_index_fixture`。
   - 复制成功 fixture 到新临时目录。
   - 用 Python 读取并覆写 `ci-artifact-index.json` 的 `commitSha` 为错误值。
   - 运行 validator，期望失败。
   - 匹配 `FAIL index commit`。
   - 清理临时目录和输出文件。
   - 增加 marker grep，锁定新负向 fixture 和失败项。
2. 更新文档：
   - README 本地验证说明补充 artifact index 身份错包负向 fixture。
   - `md/flow/flow.md` 和 `md/flow/flowchart.md` 同步 validator fixture 覆盖。
   - `md/test/test.md` smoke 基线和 validator 描述补充 artifact index 身份错包负向 fixture。
   - `update_log.md` 追加 v0.24 记录。

## 测试要求

人工明确要求“不得在本地测试，都去云端”。本轮 Agent B 不运行本地测试命令，包括 `git diff --check`、Ruby/YAML/plist 检查、`bash scripts/verify_project.sh`、本地 Xcode build 和本地 validator。

本轮验证只走云端：

- push 到 `origin/main` 触发 `.github/workflows/ci-results.yml`。
- 云端运行静态检查、`scripts/verify_project.sh`、Mac build 和 iOS generic build。
- Agent C 下载最新云端 artifact 后核对 manifest、artifact index、run context、JUnit、failure summary、主日志、Mac/iOS `.xcresult` 和 Mac 快照 manifest。

## 文档更新要求

- README 面向用户和维护者，说明验证覆盖 validator 小型成功、artifactName mismatch、artifact index 身份错包和本地缺失产物负向 fixture。
- `md/flow/flow.md` 与 `md/flow/flowchart.md` 描述当前真实验证链路。
- `md/test/test.md` 更新 smoke 当前基线，并保留人工要求云端验证时不跑本地测试的说明。
- `update_log.md` 记录版本、日期、核心变更、关键文件、验证策略和遗留事项。

## 验收标准

- 云端 `scripts/verify_project.sh` 能证明 validator 成功 fixture 放行。
- artifactName mismatch 负向 fixture 仍失败并匹配 `FAIL run context artifact name`。
- 新增 artifact index 身份错包负向 fixture 失败并匹配 `FAIL index commit`。
- 本地缺失产物负向 fixture 仍失败并匹配 `FAIL index required local artifacts`。
- GitHub Actions run 对最新 `origin/main` commit 结论为 success。
- Agent C 下载最新 run artifact，并确认 branch、commitSha、run id、run attempt 与 `origin/main` 最新状态一致。

## 风险和禁止项

- 禁止把负向 fixture 写成只要 validator 失败就通过；必须匹配目标失败项。
- 禁止修改生产 CI artifact 格式。
- 禁止运行任何本地测试命令。
- 禁止读取、提交或删除无关未跟踪文件。

## main 同步、提交和 push 要求

Agent B 开始前必须执行：

```bash
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch
```

若本轮按人工最新约束不能运行本地测试，仍需用只读状态命令确认分支和 diff 边界。提交前确认只包含本轮相关文件，提交信息使用：

```text
v0.24: 增加 CI 索引身份负向 fixture
```

推送目标固定为：

```bash
git push origin main
```

## GitHub Actions 和 Agent C 复判要求

- 触发 workflow：`.github/workflows/ci-results.yml`
- Agent C 下载 artifact 前必须先完成 `gh auth login`，并确认 GitHub 账号为项目授权账号。
- Agent C 下载最新 `origin/main` commit 对应 run 的 artifact 到 `/private/tmp/chronofocus-c-review-<run_id>/`。
- Agent C 核对 `ci-artifact-manifest.json`、`ci-artifact-index.json`、`ci-run-context.txt`、`ci-failure-summary.md`、`junit.xml`、主日志、Mac/iOS `.xcresult` 和 Mac 快照 manifest。
- Agent C 重点确认云端 `verify_project.log` 已输出 `Project structure verified.`，云端 Mac/iOS build 日志包含 `** BUILD SUCCEEDED **`，artifact index `missingRequiredCount` 为 0，且结果包身份字段完全匹配本轮 run。

# v0.68（CI 结果包名称锚点复判）

## 版本号

v0.68

## 版本分配依据

人工未指定版本号。当前最新完成版本为 v0.67，本轮继续围绕“优化 CI 测试”和云端 artifact 可靠验收做小步增强，因此自动递增为 v0.68。

## 背景

`.github/workflows/ci-results.yml` 已在 `Prepare result metadata` 阶段计算 `ARTIFACT_NAME`，并写入 `ci-run-context.txt` 与上传 artifact 名称。但 `ci-artifact-manifest.json` 当前没有 `artifactName` 字段。Agent C 和 validator 已复判 run context 的 artifact 名称，但 manifest 缺少同一个名称锚点，降低了结果包内部身份交叉核对强度。

## 目标

- 在 CI manifest 中写入 `artifactName`。
- `scripts/validate_ci_artifact.rb` 新增 `manifest artifact name` 复判，要求 manifest 中的 `artifactName` 与固定规则计算出的预期名称一致，并与 `ci-run-context.txt` 的 `artifactName` 一致。
- `scripts/verify_project.sh` 增加小型成功 fixture 的 manifest `artifactName`，并增加篡改 manifest artifactName 的负向 fixture。
- 同步 README、测试规范、核心流程文档、流程图和更新日志。

## 非目标

- 不修改 Swift 业务代码、UI、计时状态机、`FocusStore` 或平台服务。
- 不重复 v0.65 的 CI process version 复判。
- 不修改 artifact 上传名称规则，不改变 workflow 触发分支或上传内容集合。
- 不引入第三方依赖。

## 当前架构依据

- `.github/workflows/ci-results.yml` 负责生成 manifest、run context、artifact index、JUnit、failure summary 并上传 artifact。
- `scripts/validate_ci_artifact.rb` 是 Agent C 下载云端结果包后的结构化复判脚本。
- `scripts/verify_project.sh` 维护 validator 的小型成功 fixture 和负向 fixture，确保 validator 不放行自洽错包。
- 用户明确要求“不得在本地测试，都去云端”；本轮本地只允许文件读取、diff/status、commit/push、云端查询、下载云端 artifact 和对下载 artifact 运行 validator。

## 实现步骤

1. 更新 `.github/workflows/ci-results.yml`：
   - 在 manifest 中加入 `"artifactName": os.environ["ARTIFACT_NAME"]`。

2. 更新 `scripts/validate_ci_artifact.rb`：
   - 在 `expected_artifact_name` 计算后新增 `manifest artifact name` 检查。
   - 检查条件必须同时覆盖：
     - `manifest["artifactName"] == expected_artifact_name`
     - `manifest["artifactName"] == run_context["artifactName"]`

3. 更新 `scripts/verify_project.sh`：
   - 在 workflow/validator 实现标记检查中加入 manifest `artifactName` 和 `manifest artifact name`。
   - 小型成功 fixture 的 manifest 增加 `"artifactName": f"chronofocus-ci-v0.10-main-fixture-run{run_id}-attempt{attempt}"`。
   - 增加 `negative_manifest_artifact_name_fixture`，只篡改 manifest 的 `artifactName`，预期 validator 输出 `FAIL manifest artifact name`。

4. 同步文档：
   - README 描述 manifest artifactName 复判。
   - `md/test/test.md` 更新 validator 范围、负向 fixture 和 Agent C PASS 清单。
   - `md/flow/flow.md` 与 `md/flow/flowchart.md` 更新结果包字段和复判说明。
   - `update_log.md` 追加 v0.68 记录。

## 测试要求

本轮不得运行本地测试、构建、lint、`git diff --check`、`bash scripts/verify_project.sh` 或本地 fixture validator。

允许在 push 后由 Agent C 对下载的最新云端 artifact 运行：

```bash
ruby scripts/validate_ci_artifact.rb /private/tmp/chronofocus-c-review-<run_id> \
  --commit <origin-main-sha> \
  --run-id <run_id> \
  --attempt <run_attempt>
```

## 文档更新要求

- 文档必须说明 manifest 现在包含 artifactName，并由 validator 复判。
- `update_log.md` 必须记录未运行本地测试的原因和云端结论待定。

## 验收标准

- 最新云端 artifact 的 `ci-artifact-manifest.json` 包含正确 `artifactName`。
- validator 输出包含 `PASS manifest artifact name`。
- validator 仍输出既有关键 PASS，包括 `PASS run context artifact name`、`PASS ci process version`、`PASS index unexpected entries` 和 v0.67 新增的 `PASS verify_project mac mini quick panel accessibility contracts`。
- GitHub Actions 最新 `origin/main` run 成功，manifest、run context、artifact 名称、commit SHA、run id、attempt 与最新 `origin/main` 完全一致。

## 风险和禁止项

- 新 validator 会拒绝缺少 manifest artifactName 的旧 artifact，这是本轮预期效果。
- 不要 bump `CI_PROCESS_VERSION`，除非后续人工要求；本轮只加名称锚点，不重复 v0.65。
- 不要把旧 run、旧 artifact 或本地输出冒充最新云端结果。
- 不要触碰用户未跟踪文件。

## main 同步、提交和 push 要求

- 基于最新 `origin/main`，确认当前分支为 `main`。
- 只暂存本轮相关文件。
- 提交信息：`v0.68: 增强CI结果包名称复判`。
- push 到 `origin/main` 触发 `.github/workflows/ci-results.yml`。

## GitHub Actions 和 Agent C 要求

- Agent C 只下载本轮最新 run 的必要 artifact 到 `/private/tmp/chronofocus-c-review-<run_id>/`。
- 必须核对 `ci-artifact-manifest.json` 的 `artifactName`、`ci-run-context.txt` 的 `artifactName`、GitHub artifact 名称和 validator 预期名称一致。
- 若 manifest、run context、artifact 名称、commit、run id、attempt 任一不一致，本轮不通过。

# v0.69（JUnit 失败元素复判）

## 版本号

v0.69

## 版本分配依据

人工未指定版本号。当前最新完成版本为 v0.68，本轮继续围绕“优化 CI 测试”和云端 artifact 可靠验收做小步增强，因此自动递增为 v0.69。

## 背景

当前 `scripts/validate_ci_artifact.rb` 已检查 JUnit `tests="4"`、`failures="0"`、suite/classname 元数据、testcase 名称、日志路径和 outcome 文本。但 JUnit 结果还可能通过 `errors` 属性或 testcase 内 `<failure>` / `<error>` 元素表达失败。现有 validator 没有显式拒绝这些元素，结果包复判还可以更严格。

## 目标

- `.github/workflows/ci-results.yml` 生成 JUnit 时写入 `errors="0"`。
- `scripts/validate_ci_artifact.rb` 新增：
  - `junit errors`：要求 `errors == "0"`。
  - `junit failure elements`：要求所有 testcase 不含 `<failure>` 和 `<error>` 元素。
- `scripts/verify_project.sh` 更新小型成功 fixture，并增加 JUnit errors 计数和 testcase 内混入 failure/error 元素的负向 fixture。
- 同步 README、测试规范、核心流程文档、流程图和更新日志。

## 非目标

- 不修改 Swift 业务代码、UI、计时状态机、`FocusStore` 或平台服务。
- 不改变 GitHub Actions 阶段数量、artifact 上传清单或 workflow 触发规则。
- 不重复 v0.68 的 manifest artifactName 复判。
- 不引入第三方依赖。

## 当前架构依据

- `.github/workflows/ci-results.yml` 负责生成 JUnit 摘要。
- `scripts/validate_ci_artifact.rb` 是 Agent C 下载云端 artifact 后的结构化复判脚本。
- `scripts/verify_project.sh` 维护 validator 小型成功 fixture 和负向 fixture。
- 用户明确要求“不得在本地测试，都去云端”；本轮本地只允许文件读取、diff/status、commit/push、云端查询、下载云端 artifact 和对下载 artifact 运行 validator。

## 实现步骤

1. 更新 `.github/workflows/ci-results.yml`：
   - JUnit `testsuite` 写入 `errors="0"`。

2. 更新 `scripts/validate_ci_artifact.rb`：
   - 新增 `junit errors` 检查。
   - 新增 `junit failure elements` 检查，拒绝任何 testcase 下的 `failure` 或 `error` 元素。

3. 更新 `scripts/verify_project.sh`：
   - 小型成功 fixture 的 `testsuite` 增加 `errors="0"`。
   - 自检 grep 增加 `junit errors`、`junit failure elements`、对应 FAIL marker 和负向 fixture 名。
   - 增加 `negative_junit_errors_fixture`：篡改 testsuite `errors`，预期 validator 输出 `FAIL junit errors`。
   - 增加 `negative_junit_failure_element_fixture`：向某个 testcase 写入 `<failure>` 或 `<error>`，预期 validator 输出 `FAIL junit failure elements`。

4. 同步文档：
   - README 描述 JUnit `errors` 和 failure/error 元素复判。
   - `md/test/test.md` 更新 validator 范围、负向 fixture 和 Agent C PASS 清单。
   - `md/flow/flow.md` 与 `md/flow/flowchart.md` 更新结果包复判说明。
   - `update_log.md` 追加 v0.69 记录。

## 测试要求

本轮不得运行本地测试、构建、lint、`git diff --check`、`bash scripts/verify_project.sh` 或本地 fixture validator。

允许在 push 后由 Agent C 对下载的最新云端 artifact 运行：

```bash
ruby scripts/validate_ci_artifact.rb /private/tmp/chronofocus-c-review-<run_id> \
  --commit <origin-main-sha> \
  --run-id <run_id> \
  --attempt <run_attempt>
```

## 文档更新要求

- 文档必须说明 JUnit `errors` 和 testcase failure/error 元素由 validator 复判。
- `update_log.md` 必须记录未运行本地测试的原因和云端结论待定。

## 验收标准

- 最新云端 artifact 的 `junit.xml` 包含 `errors="0"`。
- validator 输出包含 `PASS junit errors` 和 `PASS junit failure elements`。
- validator 仍输出既有关键 PASS，包括 `PASS manifest artifact name`、`PASS junit metadata`、`PASS junit testcase outcomes` 和 `PASS verify_project mac mini quick panel accessibility contracts`。
- GitHub Actions 最新 `origin/main` run 成功，manifest、run context、artifact 名称、commit SHA、run id、attempt 与最新 `origin/main` 完全一致。

## 风险和禁止项

- 新 validator 会拒绝缺少 `errors="0"` 或含 failure/error 元素的旧 artifact，这是本轮预期效果。
- 不要把旧 run、旧 artifact 或本地输出冒充最新云端结果。
- 不要触碰用户未跟踪文件。

## main 同步、提交和 push 要求

- 基于最新 `origin/main`，确认当前分支为 `main`。
- 只暂存本轮相关文件。
- 提交信息：`v0.69: 增强JUnit失败元素复判`。
- push 到 `origin/main` 触发 `.github/workflows/ci-results.yml`。

## GitHub Actions 和 Agent C 要求

- Agent C 只下载本轮最新 run 的必要 artifact 到 `/private/tmp/chronofocus-c-review-<run_id>/`。
- 必须核对 `junit.xml` 的 `tests="4"`、`failures="0"`、`errors="0"`、四个 testcase、outcome 文本和无 failure/error 元素。
- 若 manifest、run context、artifact 名称、commit、run id、attempt 任一不一致，本轮不通过。

# v0.64（iOS新增入口分类上下文）

## 版本号

v0.64

## 版本分配依据

人工总目标要求持续优化 UI 易用性、分类体验和 CI 测试。v0.63 已完成计划面板生成/清空操作语义并通过云端 artifact 验收；本轮继续按小版本递增到 v0.64，聚焦 iOS 日程页“新增待办”入口在选中分类筛选时的操作语义。

## 背景

iOS 日程页右上角 `+` 按钮会打开 `TaskEditorView`，并在存在 `selectedCategory` 时把当前分类作为新待办的初始分类。但按钮本身当前只读作“新增待办”，没有说明会预填当前分类，也缺少按分类发起新增的 Voice Control input labels。用户在筛选分类后新增待办时，辅助技术无法直接确认“新增此分类待办”这一上下文。

## 目标

- iOS 日程页 toolbar “新增待办”按钮在未选中分类时保持通用语义。
- iOS 日程页 toolbar “新增待办”按钮在选中分类时读出“新增当前分类待办”。
- 为该按钮补齐可访问 hint，说明选中分类时会预填当前分类。
- 为该按钮补齐 Voice Control input labels，支持“新增待办”“新增此分类”“新增某分类分类待办”等入口。
- 在 `scripts/verify_project.sh` 增加源码契约 marker，并让 `scripts/validate_ci_artifact.rb` 复判该 marker。
- 同步 README、测试规范、核心流程、流程图和更新记录。

## 非目标

- 不改 `TaskEditorView` 表单、保存逻辑、分类清洗、`FocusStore`、`TimerEngine` 或计划生成行为。
- 不改变 toolbar 可见按钮文案或布局。
- 不同步修改 macOS 快速新增；该候选保留给后续轮次。
- 不运行本地测试、build、lint、`git diff --check` 或脚本验证；人工已要求“不得在本地测试，都去云端”。

## 当前架构依据

- iOS 日程页位于 `ChronoFocus/Views/ScheduleView.swift`。
- `ScheduleView` 已用 `selectedCategory` 控制任务列表筛选，并在 `.sheet(isPresented: $showingEditor)` 中把 `selectedCategory` 传给 `TaskEditorView(initialDueDate:initialCategory:)`。
- 可访问语义应留在 SwiftUI View 层，不需要新增共享模型字段。
- `scripts/verify_project.sh` 是项目专属验证入口；云端 workflow 会运行它并把 `verify_project.log` 纳入 artifact。

## 实现步骤

1. 同步最新 `main`，确认当前分支为 `main`，没有无关 tracked diff；不要读取或提交用户未跟踪文件。
2. 修改 `ChronoFocus/Views/ScheduleView.swift`：
   - 增加 `addTaskAccessibilityLabel` helper。
   - 增加 `addTaskAccessibilityHint` helper。
   - 增加 `addTaskInputLabels` helper。

   - 在 iOS 日程源码契约中检查 helper、toolbar label、hint 和 input labels。

   - 增加缺失 marker 负向 fixture：`negative_schedule_toolbar_add_marker_fixture`。
4. 修改 `scripts/validate_ci_artifact.rb`：
   - 复判 `verify_project.log` 中的新增 marker。
   - PASS 名称为 `verify_project schedule toolbar add category context contracts`。
5. 同步 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。
